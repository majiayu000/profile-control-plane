# Evidence and positioning

Use this reference to turn repository facts into a credible profile direction without inventing identity,
impact, or intent.

## Evidence classes

| Class                    | Examples                                                                    | Allowed use                                                         |
| ------------------------ | --------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Verified fact            | Repository metadata, README claims, releases, license, public activity      | State directly and preserve the source meaning                      |
| Supported interpretation | Several maintained CLI repositories suggest a tooling focus                 | Present as a design hypothesis and cite the supporting repositories |
| User-authored intention  | Target audience, career direction, desired identity, current private work   | Use only after the user states or approves it                       |
| Unsupported claim        | Unpublished metrics, inferred employment, invented users, fabricated impact | Omit                                                                |

Treat repository descriptions as claims made by their owners, not independent proof of adoption or impact.
Use stars, releases, contributors, and recent activity only when they are relevant and current.

## Evidence inventory

Record the minimum inventory needed to explain each visible profile claim:

| Proposed claim or role               | Supporting source                        | Evidence class           | Confidence | Profile use         |
| ------------------------------------ | ---------------------------------------- | ------------------------ | ---------- | ------------------- |
| Example: Builds agent infrastructure | Three maintained agent-tool repositories | Supported interpretation | High       | Headline hypothesis |

Do not assign a semantic label such as `PLATFORM`, `PRODUCTION`, or `SECURITY` from a repository name alone.
Read the repository README and relevant documentation first. Leave uncertain descriptions blank.

## Design brief

Before editing `profile.yaml`, produce this brief in the conversation:

```text
Audience:
One-sentence promise:
Primary archetype:
Secondary influence, if needed:
Three strongest proofs:
Information order:
Visual direction:
Content to omit:
Open questions or evidence gaps:
```

Recommend one direction. Offer one alternative only when a materially different story has comparable
evidence. Do not make the user choose among cosmetic variations that have the same narrative.

## Positioning rules

- Lead with current, maintained work instead of a complete career history.
- Make the headline specific enough to predict the flagship repositories beneath it.
- Select up to three primary proofs before expanding into supporting modules.
- Prefer concrete nouns and verbs over claims such as "passionate", "innovative", or "world-class".
- Distinguish what the user builds from the technologies used to build it.
- Preserve meaningful personality when the evidence or user intent supports it; factual does not mean generic.
- Ask for approval when a choice changes the apparent profession, audience, product category, or level of
  authority.
- Keep no-data states blank or omit the section; never fill space with guessed content.
